# Queue-Driven Research Worker

NotebookLM is no longer the driver for the 50-state/local filing research program. It can still summarize official-source bundles, but the durable work queue lives in SOSFiler.

## Files

- `backend/research_worker.py` converts the state and local research plans into executable research jobs.
- `data/research_jobs.json` is the persistent job queue generated by the worker.
- `data/research_source_seeds.json` stores official-source starting points for each priority state.
- `data/research_runs/` stores source-check run artifacts.

## Commands

Initialize or refresh the queue:

```bash
python3 backend/research_worker.py init
```

Show coverage:

```bash
python3 backend/research_worker.py report
```

List the next pending jobs:

```bash
python3 backend/research_worker.py next --limit 10
```

Run a small source-seed check:

```bash
python3 backend/research_worker.py run --limit 3
```

Limit to one state:

```bash
python3 backend/research_worker.py next --state TX --limit 5
python3 backend/research_worker.py run --state TX --limit 2
```

After an operator or research agent extracts enough official evidence:

```bash
python3 backend/research_worker.py set-status ready_for_review tx-state-filing-map --note "Official fee and portal sources collected; extraction ready for review."
```

## Status Rules

- `pending`: job exists, no source pass started.
- `in_progress`: official seed sources are reachable or extraction has started.
- `blocked`: no official seed source is reachable or the job needs new source seeds.
- `ready_for_review`: official evidence has been extracted but not yet productized.
- `verified`: official evidence is sufficient for customer-facing pricing/process logic.

Only `verified` jobs should feed automatic pricing or filing promises.

## Operating Model

1. Work state by state through the Phase 1 list: AZ, TX, NV, CA, DE, FL, WY, NY, GA, IL.
2. Complete state filings, state tax/business gateways, county directories, municipality directories, county/local filings, city/local filings, and productization jobs for each state.
3. Treat these as mandatory nationwide service lanes in state filing maps: name reservation, foreign qualification/withdrawal, amendments, reinstatements, dissolution, certificates of good standing/certified copies, apostille/authentication, BOI touchpoints, payroll tax registration, and sales/use tax registration.
4. Record official URLs, forms, fees, processing fees, prerequisites, portal steps, customer options, evidence gates, and confidence levels.
5. For every filing lane, capture official cost components and compute SOSFiler price as `official cost + $9.00`.
6. Convert verified outputs into `data/filing_actions.json` records and product/operator flows.
7. Use NotebookLM only for small, deliberate official-source bundles where summarization helps.

## NotebookLM Role (Optional Sidecar)

NotebookLM can speed extraction when you have a tight official-source packet for one state/service lane. Keep it bounded:

- One state + one service lane per task (for example, `CA certificates of status and certified copies`).
- Only official source documents/URLs in the source set.
- Output must still be validated and captured back into queue evidence before status can move to `verified`.
